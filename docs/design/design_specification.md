# 시냅스노트 디자인 스펙

## 1. 문서 목적

이 문서는 `docs/design` 폴더 내 시안 이미지와 `docs/design/synapse_neural_os/DESIGN.md`를 기준으로, 시냅스노트의 비주얼 언어와 화면 구조, 컴포넌트 규칙, 반응형 패턴을 하나의 디자인 스펙으로 정리한 문서다.

이 스펙의 목적은 개별 시안을 나열하는 것이 아니라, 여러 시안에서 반복적으로 등장하는 공통 시스템을 추출해 실제 제품 디자인 기준으로 정제하는 데 있다.

## 2. 디자인 지향점

시냅스노트의 디자인 지향점은 `Digital Synapse`, `Neural Operating System`, `Thought-to-Knowledge Interface`다.

사용자는 단순히 노트를 저장하거나 채팅을 주고받는 것이 아니라, 자신의 생각이 AI와 결합되어 지식으로 연결되고 증폭되는 환경에 들어와야 한다. 따라서 전체 경험은 전통적인 문서 앱보다 더 유기적이고, 더 공기감이 있으며, 더 살아 있는 시스템처럼 느껴져야 한다.

핵심 감정은 다음과 같다.

- 정적 문서 툴이 아닌 살아 있는 지식 시스템
- 평면적인 업무 UI가 아닌 깊이감 있는 인지 작업 공간
- 기계적 클릭이 아닌 신호 흐름 같은 상호작용

## 3. 경험 원칙

### 3.1 Context First

모든 주요 화면은 현재 AI가 무엇을 보고 있는지 드러내야 한다. 오른쪽 `Active Context` 레일 또는 모바일 컨텍스트 화면은 보조 패널이 아니라 제품의 핵심 축이다.

### 3.2 Graph Is a View, Base Is the System

`Knowledge Base`는 저장소 전체를 뜻하고, `Knowledge Graph View`는 그 저장소를 시각적으로 탐색하는 대표 뷰다. 따라서 그래프는 단독 기능이 아니라 라이브러리, 채팅, 문서 편집과 이어지는 전체 흐름의 일부여야 한다.

### 3.3 Capture Must Feel Immediate

대화 중 생성된 내용을 저장하는 순간은 별도 모달 중심이 아니라 대화 흐름 안에서 자연스럽게 일어나야 한다. `Saved as`, `Agent Action`, `Capture` 같은 피드백은 빠르고 명료해야 한다.

### 3.4 Focused Density

전체 미학은 미니멀하지만 정보량은 적지 않다. 텍스트, 메타데이터, 프리뷰, 상태값을 한 화면에 담되 선 대신 톤, 간격, 발광, 투명도를 사용해 계층을 나눈다.

### 3.5 One Shell, Multiple Modes

채팅, 라이브러리, 그래프, 편집기, 학습 모드는 서로 다른 앱처럼 보이면 안 된다. 하나의 제품 셸 안에서 작업 모드만 바뀌는 느낌을 유지해야 한다.

## 4. 비주얼 시스템

### 4.1 컬러 팔레트

공통적으로 확인되는 주요 컬러는 다음과 같다.

- Background / Void: `#131313`
- Primary Text Accent: `#D2BBFF`
- Primary Container: `#7C3AED`
- Secondary Accent: `#3B82F6` 또는 이에 준하는 블루 계열
- Secondary Text Accent: `#ADC6FF`
- Warm Highlight: `#FFB784`
- Border / Outline: 흰색 또는 보라 계열의 저투명도 색상

색상 사용 원칙은 다음과 같다.

- 기본 배경은 순수 검정이 아니라 깊이감 있는 차콜을 사용한다.
- 인터랙션 포인트는 보라-블루 축으로 통일한다.
- 강조가 필요한 문맥 정보나 인사이트에는 앰버 계열을 보조적으로 사용한다.
- 선명한 솔리드 경계 대신 투명도와 발광으로 시선을 유도한다.

### 4.2 표면과 질감

시안 대부분은 `다크 배경 + 유리 패널 + 저투명 경계 + 은은한 글로우` 조합을 사용한다.

기본 규칙은 다음과 같다.

- 주요 패널: 반투명 다크 서피스 + `backdrop blur`
- 구획 분리: 1px 실선보다 톤 차이와 간격 우선
- 선택 상태: 외곽선보다 발광 오라와 그라디언트 강조 우선
- 떠 있는 패널: 유리감 있는 배경과 부드러운 그림자 사용

### 4.3 타이포그래피

공통 폰트 방향은 다음과 같다.

- 헤드라인: `Lexend`
- 본문: `Inter`
- 메타데이터/라벨: `Inter`

타이포 원칙은 다음과 같다.

- 제품명, 대화면 제목, 탭 헤더는 Lexend 사용
- 본문과 설명문은 Inter 사용
- 라벨은 대문자 또는 넓은 자간으로 시스템적 느낌 강화
- 제목과 메타데이터 간 크기 대비를 크게 두어 에디토리얼한 위계 형성

### 4.4 모서리와 형태

- 기본 카드: `16px`~`24px` 수준의 둥근 모서리
- CTA 버튼: `12px`~`16px` 수준의 라운드
- 노드 카드: 문서형은 라운드 사각형, 개념형은 원형
- 모바일 하단 내비게이션: 상단보다 더 큰 라운드 처리

## 5. 공통 레이아웃 셸

### 5.1 데스크톱 셸

데스크톱 시안의 공통 구조는 다음과 같다.

1. 좌측 고정 네비게이션 레일
2. 상단 유틸리티 바
3. 중앙 메인 워크스페이스
4. 우측 `Active Context` 또는 인사이트 레일

이 구조는 채팅, 그래프, 문서 편집기, 학습 모드에서 반복된다. 따라서 실제 구현에서도 화면마다 레이아웃을 새로 만드는 대신 공통 앱 셸을 기준으로 각 모드 콘텐츠만 바뀌는 방식이 적합하다.

### 5.2 모바일 셸

모바일 시안은 데스크톱 구조를 단순 축소하지 않고 다음 방식으로 재구성한다.

- 상단 앱 바
- 중간 모드 탭 또는 현재 화면 제목
- 세로 스크롤 가능한 메인 콘텐츠
- 하단 고정 내비게이션

여기서 중요한 점은 `Chat / Context / Graph`의 3축이 모바일에서도 유지된다는 것이다. 사용자는 탭 또는 좌우 전환을 통해 같은 세션 상태를 유지한 채 화면을 넘나들 수 있어야 한다.

## 6. 핵심 화면군 스펙

### 6.1 Agent Chatting Workspace

주요 특징:

- 중앙 대화 스레드 중심 레이아웃
- 우측 `Active Context` 레일 고정
- 하단 입력창은 넓고 낮은 옴니바 형태
- 대화 안에서 저장 완료 카드, 에이전트 액션, 인사이트 블록 표시
- 상단에서 사용 중인 모델을 인지 가능하게 표시

디자인 원칙:

- 사용자 메시지와 에이전트 메시지는 말풍선이 아니라 패널형 카드로 다룬다.
- 에이전트 응답 내부에 `Neural Insight`, `Agent Action`, `Saved as Knowledge Node` 같은 구조화된 보조 블록을 삽입할 수 있어야 한다.
- 컨텍스트가 연결된 세션임을 상단 배지, 로딩 바, 상태 라벨로 지속적으로 드러낸다.

### 6.2 Context Manager / Context Rail

주요 특징:

- 대부분의 데스크톱 시안에서 우측 사이드 레일로 표현
- 모바일에서는 별도 전환 화면으로 확장
- 노드별 타입, 파일명, 메타데이터, 토글 상태 제공
- 드래그 앤 드롭 주입 구역 존재
- 컨텍스트 준비 상태와 다음 액션 CTA 표시

디자인 원칙:

- 단순 체크리스트가 아니라 현재 AI 세션의 작동 조건을 설정하는 제어면처럼 보여야 한다.
- 각 컨텍스트 아이템은 아이콘, 제목, 서브텍스트, 활성 토글을 가져야 한다.
- 하단에는 `Switch to Chat`, `Context Ready`, `Neural Load`, `Merge`, `Inject` 같은 세션 전환 행동을 배치한다.

### 6.3 Knowledge Base Library View

주요 특징:

- 카드형 콘텐츠 그리드
- 상단 검색, 필터, 뷰 전환, Export 액션
- 외부 AI/서비스 연동 상태를 보여주는 Integration 블록
- 문서 타입 태그, 날짜, 설명 요약 제공

디자인 원칙:

- 라이브러리 뷰는 Knowledge Base의 탐색 입구다.
- Graph View보다 더 정돈된 목록 중심 화면으로 구성한다.
- 외부 소스에서 유입된 지식과 내부 문서를 동일한 디자인 언어 안에서 분류한다.

### 6.4 Knowledge Graph View

주요 특징:

- 어두운 보이드 배경 위 점형 또는 입자형 그리드
- 루트에서 하위 노드로 이어지는 연결선
- 활성 노드 주변 발광 강조
- 선택 노드의 확장 카드 프리뷰
- 우측 컨텍스트 레일과 하단 툴바 동시 사용

디자인 원칙:

- 그래프는 전체를 한 번에 읽는 지도보다 현재 노드를 중심으로 탐색하는 인터페이스여야 한다.
- 선택된 노드는 크게 떠오르고, 연결 노드는 다음 행동이 가능할 정도의 정보 밀도로 보여야 한다.
- 연결선은 지나치게 진하지 않게 유지하되 활성 연결에는 발광 또는 흐르는 듯한 강조를 준다.

### 6.5 Document Editor View

주요 특징:

- 중앙 대형 읽기/편집 영역
- 좌측 컬렉션 네비게이션
- 우측 활성 컨텍스트 레일
- 본문 안에 코드 블록, 메타데이터, AI 보조 툴바 사용

디자인 원칙:

- 에디터는 일반 마크다운 편집기보다 더 큰 제목과 넓은 본문 폭으로 에디토리얼한 인상을 준다.
- 편집 도중에도 현재 문서가 어떤 컨텍스트와 연결되는지 우측 레일에서 확인 가능해야 한다.
- AI 보조 툴은 문서 흐름을 깨지 않도록 본문 하단 또는 선택 영역 인접 위치에 노출한다.

### 6.6 Learning Studio

주요 특징:

- 중앙의 질문 프롬프트와 답변 작성 영역
- 우측 `Knowledge Gaps` 패널
- `Get a Hint`, `Deepen the Topic`, `Review Schedule` 등 학습 액션
- 모드 탭: `Socratic Mode`, `Review Mode`

디자인 원칙:

- 학습 모드는 시험 화면보다 사유 공간처럼 보여야 한다.
- 사용자가 쓰는 캔버스는 충분히 넓고 방해 요소가 적어야 한다.
- 우측 패널은 점수판이 아니라 이해 공백을 알려주는 지적 피드백 패널로 동작해야 한다.

## 7. 내비게이션 패턴

### 7.1 좌측 레일

좌측 레일은 거의 모든 데스크톱 시안에서 유지된다. 기본 항목은 다음 축으로 정리할 수 있다.

- Chat
- Graph
- Library
- Learn
- Sync / Vault / Settings
- `New Thought` CTA

원칙:

- 제품 전역 이동과 새 작업 생성이 같은 레일 안에 있어야 한다.
- 현재 활성 화면은 보라 계열 배경, 광원 효과, 좌측 강조선 등으로 분명히 구분한다.

### 7.2 상단 바

상단 바에는 다음 요소가 반복된다.

- 제품명 또는 현재 모드명
- 모델 전환 탭
- 검색 입력창
- 알림/설정/프로필 아이콘
- 주요 액션 버튼

원칙:

- 상단 바는 정보량이 많지만 무거워 보이면 안 된다.
- 반투명 서피스와 얇은 구분선으로 워크스페이스 위에 떠 있는 느낌을 유지한다.

### 7.3 하단 바와 모바일 내비게이션

모바일 하단 바는 `Chat / Context / Graph` 3항 구조가 기본이다.

원칙:

- 활성 탭은 글로우가 있는 캡슐 또는 라운드 직사각형으로 표시
- 비활성 탭은 낮은 채도와 낮은 불투명도 유지
- 입력창과 하단 내비게이션은 시각적으로 충돌하지 않도록 충분한 간격 확보

## 8. 컴포넌트 스펙

### 8.1 버튼

- Primary CTA: 보라-블루 그라디언트, 글로우, 라운드 모서리
- Secondary CTA: 어두운 유리 패널 위 라벨 강조
- Utility Button: 아이콘 중심, 낮은 채도, hover 시 배경 토닝

### 8.2 입력창

- 메인 입력은 `Inject new thoughts...`, `Neural command...`, `Search...` 같은 의미 중심 플레이스홀더 사용
- 바깥선보다 배경/글로우/내부 패딩으로 질감 형성
- 채팅 입력창은 첨부, 음성, 전송 같은 보조 액션을 붙일 수 있어야 함

### 8.3 카드

- 문서 카드: 제목, 날짜, 요약, 태그
- 컨텍스트 카드: 아이콘, 이름, 유형, 토글
- 인사이트 카드: 짧은 요약, 강조 색, 다음 행동 CTA
- 노드 프리뷰 카드: 제목, 타입, 미리보기, 액션 버튼

### 8.4 노드

- Concept Node: 원형, 작은 라벨, 색상 기반 주제 구분
- Content Node: 라운드 사각형, 아이콘, 제목, 프리뷰
- Active Node: 더 강한 외곽 글로우, 배경 농도 상승, 연결선 강조
- Drop Target: 점선 박스 또는 약한 보라 발광으로 주입 가능 상태 표시

## 9. 인터랙션 및 모션

공통 모션 원칙은 `짧고 빠른 반응`보다 `부드럽고 길게 이어지는 흐름`에 가깝다.

- 기본 전환: `300ms`~`500ms`
- 이징: `cubic-bezier(0.4, 0, 0.2, 1)` 계열
- hover: 배경 톤 변화 + 미세 확대 + 글로우 강화
- 선택: 발광 오라, 외곽선 강조, 연결선 활성화
- 드래그: 드롭 존 활성화, 연결 문맥 강조

지양할 것은 다음과 같다.

- 과도한 바운스
- 잦은 깜빡임
- 화면 전체를 흔드는 과한 패럴랙스

## 10. 반응형 규칙

### 10.1 데스크톱

- 좌측 레일 고정
- 우측 컨텍스트 레일 표시
- 중앙 화면은 모드별로 최대 폭과 정보 밀도 조절

### 10.2 태블릿

- 좌측 레일 축소 가능
- 우측 레일은 접거나 오버레이 전환 가능
- 그래프와 채팅 간 전환은 유지하되 한 번에 보여주는 패널 수 축소

### 10.3 모바일

- `Chat / Context / Graph` 흐름을 중심으로 재구성
- Context는 별도 화면으로 확장
- 하단 고정 내비게이션 유지
- 대형 드래그 앤 드롭보다 탭/추가/제거 중심 인터랙션 사용

## 11. 접근성 및 사용성 가이드

- 본문 대비는 최소 `4.5:1` 이상 유지
- 발광 효과만으로 상태를 구분하지 않고 텍스트/아이콘/배지도 함께 사용
- 토글과 CTA는 터치 환경에서 충분한 크기를 확보
- 그래프 노드의 선택 상태는 색뿐 아니라 크기, 외곽선, 라벨로 중복 표현
- 장시간 사용을 고려해 지나치게 강한 네온 대비는 피하고, 핵심 요소에만 집중 사용

## 12. 실제 제품 반영 원칙

현재 시안은 매우 강한 콘셉트 렌더링 성격을 가진다. 실제 제품 반영 시에는 다음처럼 정제하는 것이 적절하다.

1. 콘셉트는 유지하되 장식적 요소는 줄인다.
2. `Knowledge Base`, `Context Manager`, `Agent Chatting`의 구조를 우선시한다.
3. 글래스모피즘은 선택적 강조에 사용하고, 본문 가독성을 해치지 않도록 제한한다.
4. 보라-블루 계열은 제품 정체성으로 유지하되 정보 계층은 중립색으로 안정화한다.
5. 그래프, 라이브러리, 편집기, 학습 모드가 하나의 시스템처럼 느껴지도록 공통 셸을 유지한다.
