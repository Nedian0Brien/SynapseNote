# 시냅스노트 통합 기획서

## 1. 문서 목적

이 문서는 `docs/specification.md`와 `docs/prompt_memo.md`의 내용을 통합하여 시냅스노트의 제품 비전, 핵심 사용자 경험, 화면 구조, 노드 시스템, 기술 방향을 하나의 기준 문서로 정리한 기획서다.

시냅스노트는 단순한 노트 앱이나 단순한 AI 채팅 앱이 아니라, 대화형 AI 경험을 장기적으로 축적 가능한 개인 지식 자산으로 전환하는 AI 에이전트 기반 지식 운영체제를 목표로 한다.

## 2. 제품 정의

### 2.1 한 줄 정의

시냅스노트는 AI와의 대화, 문서 편집, Knowledge Base, 컨텍스트 제어를 하나의 흐름으로 연결한 개인용 에이전틱 지식 베이스다.

### 2.2 해결하려는 문제

기존 노트 앱은 저장과 구조화에는 강하지만 대화형 활용과 지식 회수 경험이 약하다. 반대로 기존 챗봇은 생성과 질의응답에는 강하지만, 대화 중 나온 지식이 장기 자산으로 축적되지 않고 쉽게 휘발된다.

사용자는 점점 더 많은 정보를 AI와 대화로 얻고 있지만, 그 내용은 이후 검색, 학습, 재사용, 맥락 연결이 어렵다. 시냅스노트는 이 단절을 해결해 대화를 지식으로 남기고, 남겨진 지식을 다시 대화와 학습에 활용할 수 있게 한다.

### 2.3 제품 목표

1. AI와 대화하면서 지식을 실시간으로 축적한다.
2. 축적된 지식을 AI가 다시 검색, 조합, 설명할 수 있게 한다.
3. 사용자가 AI에 주입되는 컨텍스트를 직접 선택하고 통제할 수 있게 한다.
4. 장기적으로는 사용자의 사고 체계를 보조하는 세컨드 브레인으로 발전시킨다.

## 3. 핵심 가치 제안

### 3.1 Capture

AI와 대화하면서 생성된 내용을 즉시 저장 가능한 지식 단위로 전환한다.

- "방금 대화한 논문 요약 저장해줘"
- "이 아이디어를 프로젝트 X 지식으로 넣어줘"
- "이 답변을 내 영어 표현 노트로 축적해줘"

### 3.2 Recall

저장된 지식을 자연어 대화로 다시 불러오고 재조합한다.

- "지난달 정리한 RAG 평가 기준 다시 보여줘"
- "벡터 검색 관련 내용만 모아줘"

### 3.3 Context Control

에이전트가 어떤 문서, 노드, 프로젝트, 기간, 주제를 참고하는지 사용자가 직관적으로 통제한다.

- 현재 대화에 주입된 컨텍스트를 시각적으로 확인
- 특정 프로젝트나 노드만 포함 또는 제외
- 드래그, 다중 선택, 토글 기반 주입

### 3.4 Learn

저장된 지식을 단순 보관이 아니라 학습 가능한 형태로 재구성한다.

- 소크라테스식 질의
- 간격 반복
- 개념 연결 퀴즈
- 취약 개념 중심 복습

## 4. 제품 핵심 개념

시냅스노트의 핵심은 `Agent Chatting`, `Context Manager`, `Knowledge Base`의 3개 축이다.

### 4.1 Agent Chatting

사용자가 AI 에이전트와 대화하며 질문, 생성, 요약, 탐색, 회수를 수행하는 공간이다. 단순 채팅창이 아니라 지식 생성의 출발점이며, 채팅 내용은 적극적으로 노드화되어 장기 자산으로 편입되어야 한다.

### 4.2 Context Manager

현재 대화에 어떤 맥락이 주입되는지 시각적으로 관리하는 브릿지 레이어다. Agent Chatting과 Knowledge Base 사이에 위치하며, 사용자가 노드나 문서를 선택해 현재 대화 컨텍스트에 넣거나 뺄 수 있어야 한다.

### 4.3 Knowledge Base

사용자의 영구 지식 저장소다. 이 영역의 상위 개념은 Knowledge Base이며, 대표적인 진입 방식은 카드 중심의 `Library View`와 노드 중심의 `Knowledge Graph View`다. 사용자는 이 베이스 안에서 개념, 문서, 대화, 프로젝트 자산 간 관계를 저장, 탐색, 편집, 재사용한다.

## 5. 정보 구조와 화면 원칙

### 5.1 기본 화면 구조

세 화면은 수평적으로 연결되어 있어야 한다.

1. `Chat + Context Manager` 화면
2. `Context Manager + Knowledge Base` 화면

Context Manager는 항상 중심에 위치하며, 사용자는 이를 축으로 좌우 화면을 자연스럽게 이동한다. 구조적으로는 Chat과 Knowledge Base가 직접 분리된 것이 아니라, Context Manager를 중심으로 연결된 양쪽 영역이다.

### 5.2 모바일 전환 방식

모바일에서는 좌우 스크롤 또는 스와이프 기반 전환이 핵심이다.

- 왼쪽 방향: `Chat + Context Manager`
- 오른쪽 방향: `Context Manager + Knowledge Base`

이 구조를 통해 사용자는 작은 화면에서도 컨텍스트 관리 흐름을 유지한 채 대화와 지식 탐색을 오갈 수 있어야 한다. Context Manager는 두 화면에 공통으로 존재해야 하며, 모바일에서 정보 단절이 생기지 않는 것이 중요하다.

### 5.3 데스크톱 전환 원칙

데스크톱에서는 더 넓은 영역을 활용해 다중 패널 구성이 가능하다. 다만 모바일과 동일한 개념 구조를 유지해야 한다.

- Chat, Context Manager, Knowledge Base를 병렬 패널로 보거나
- Context Manager를 고정 중심 패널로 두고 좌우 영역을 확장하는 방식

핵심은 화면 배치가 달라져도 "컨텍스트 매니저가 브릿지 역할을 한다"는 정보 구조가 유지되는 것이다.

### 5.4 핵심 화면군

디자인 시안 기준으로 시냅스노트는 다음 6개의 화면군으로 정리할 수 있다.

1. Agent Chatting Workspace
2. Context Manager / Active Context Rail
3. Knowledge Base Library View
4. Knowledge Base `Knowledge Graph View`
5. Document Editor View
6. Learning Studio

즉, 제품 구조는 단순히 `Chat / Context / Graph` 세 화면으로 끝나지 않는다. 사용자는 동일한 앱 셸 안에서 라이브러리 탐색, 그래프 탐색, 문서 편집, 학습까지 자연스럽게 이어가야 한다.

## 6. 주요 사용자 시나리오

### 6.1 대화에서 지식 축적

1. 사용자가 에이전트와 대화한다.
2. 중요한 응답이나 사용자 입력을 선택한다.
3. 해당 내용을 문서 또는 개념 노드로 저장한다.
4. 저장된 노드는 Knowledge Base에 반영되며, 필요하면 Knowledge Graph View에서도 바로 탐색할 수 있다.
5. 이후 다른 대화에서 해당 노드를 다시 컨텍스트로 주입할 수 있다.

### 6.2 그래프에서 컨텍스트 주입

1. 사용자가 Knowledge Base의 `Knowledge Graph View`에서 관련 노드를 탐색한다.
2. 단일 또는 다중 선택으로 노드를 고른다.
3. 드래그 또는 선택 액션으로 Context Manager에 추가한다.
4. 에이전트는 현재 선택된 컨텍스트만 기준으로 응답한다.

### 6.3 노드 내부 문서 편집

1. 사용자가 그래프에서 문서형 노드를 클릭한다.
2. 해당 노드의 상세 편집 화면으로 진입한다.
3. 마크다운 기반으로 문서를 읽고 편집한다.
4. 변경 내용은 다시 그래프와 컨텍스트 흐름에 반영된다.

### 6.4 학습 모드 활용

1. 사용자가 특정 지식 묶음 또는 노드군을 선택한다.
2. AI가 이를 기반으로 질문, 퀴즈, 요약, 회고를 생성한다.
3. 사용자는 자신의 약한 개념을 확인하고 재학습한다.

## 7. Knowledge Base 및 Graph View 상세 기획

### 7.1 Knowledge Base 구조 원칙

Knowledge Base는 크게 `Library View`와 `Knowledge Graph View`를 가진다. `Library View`는 카드와 메타데이터 중심의 정돈된 탐색 화면이고, `Knowledge Graph View`는 연결 관계를 중심으로 지식을 탐색하는 시각화 화면이다. 이 중 `Knowledge Graph View`는 기본적으로 디렉토리 구조를 따른다. 루트 디렉토리부터 리프 노드까지 연결된 기본 골격이 먼저 존재하고, 그 위에 의미적 연결 관계가 추가될 수 있다.

즉, 시냅스노트의 Knowledge Base는 완전히 자유로운 비정형 그래프가 아니라 다음 두 층을 함께 가진다.

1. 파일/디렉토리 기반의 구조적 연결
2. 개념, 주제, 출처, 대화 맥락 기반의 의미적 연결

### 7.2 노드 선택과 포커스

노드를 클릭하면 해당 노드는 선택 상태가 되며 화면 중앙으로 이동해야 한다. 동시에 해당 노드와 연결된 주변 노드들은 사용자가 다음 액션을 취하기 좋은 형태로 재배열되어야 한다.

핵심은 "그래프 전체를 한 번에 읽게 하는 것"이 아니라, 현재 선택된 노드를 중심으로 국소 탐색 경험을 강화하는 것이다.

### 7.3 연결 노드 정보 표현

현재 선택된 노드와 연결된 노드 중 내용이 있는 노드는 제목과 함께 짧은 내용 프리뷰를 노출해야 한다. 사용자는 이 프리뷰만으로도 다음에 어떤 노드를 열지 판단할 수 있어야 한다.

프리뷰는 정보량이 과하지 않게 제한하되, 제목만 보이는 상태보다 탐색 효율이 높아야 한다.

### 7.4 다중 선택과 컨텍스트 주입

그래프 노드는 단일 클릭 탐색만 지원해서는 안 된다. 다중 선택, 드래그, 일괄 추가 등 컨텍스트 주입 중심 상호작용을 지원해야 한다.

지원해야 하는 대표 액션은 다음과 같다.

- 단일 선택
- 다중 선택
- 드래그하여 Context Manager로 이동
- 선택된 노드 일괄 컨텍스트 추가
- 선택된 노드 일괄 제외

### 7.5 Graph View 시각 규칙

디자인 시안 기준으로 `Knowledge Graph View`는 단순 노드 캔버스가 아니라 집중형 탐색 인터페이스여야 한다.

- 배경은 점형 그리드나 미세 입자가 있는 다크 보이드
- 활성 노드는 발광 오라와 확장 카드로 강조
- 연결선은 평상시에는 약하게, 활성 상태에서는 더 선명하게 표현
- 우측에는 `Active Context` 레일을 함께 두어 그래프 탐색과 컨텍스트 주입을 하나의 흐름으로 묶음
- 하단에는 줌, 포커스, 필터, 모드 전환 같은 그래프 전용 툴바 배치

## 8. 노드 시스템 기획

### 8.1 노드 유형

시냅스노트의 노드는 단순 문서가 아니라 다양한 지식 자산을 표현해야 한다.

- Concept / 개념어
- Directory
- 일반 문서
- 사람이 편집한 문서
- 에이전트 채팅
- 에이전트 생성 문서
- Code
- Git Issue
- Web Article
- PDF 문서

향후 외부 서비스 연동이 확장되더라도 같은 모델 안에 수용될 수 있어야 한다.

### 8.2 형태 규칙

- 내용이 없는 개념 노드: 원형
- 내용이 있는 노드: 둥근 모서리의 사각형

이 규칙은 사용자가 한눈에 "개념 중심 노드인지, 실제 내용을 가진 자산인지"를 구분하게 해준다.

### 8.3 확장 상태

내용이 있는 사각형 노드는 확장/축소 상태를 가져야 한다.

- 축소 상태: 제목과 타입 중심
- 확장 상태: 제목, 타입, 짧은 미리보기

노드 확장은 그래프 가독성을 해치지 않으면서도 정보 밀도를 높이는 수단이어야 한다.

### 8.4 시각적 구분 원칙

노드 유형에 따라 사용자가 직관적으로 종류를 식별할 수 있어야 한다. 이를 위해 아이콘, 컬러, 배지, 메타데이터 패턴을 함께 사용한다.

예시:

- Directory: 폴더 아이콘, 구조적 연결 강조
- Agent Chat: 대화 아이콘, 발화 수 또는 최근 갱신 시간 노출
- PDF: 문서 아이콘, 페이지 수 또는 요약 프리뷰
- Code: 코드 아이콘, 언어 정보 표시
- Git Issue: 이슈 상태 배지, 번호 표시
- Web Article: 링크/브라우저 아이콘, 출처 도메인 표시

## 9. Context Manager 상세 기획

Context Manager는 단순 선택 목록이 아니라, 현재 AI가 참조 중인 지식 집합을 시각적으로 조작하는 핵심 허브다.

필수 역할은 다음과 같다.

1. 현재 주입된 노드 목록 표시
2. 각 노드의 출처와 타입 표시
3. 포함/제외/잠금 상태 제어
4. Agent Chatting과 Knowledge Base 사이의 브릿지 역할 수행
5. 다중 선택된 노드의 일괄 주입 처리

이 영역은 사용자가 "AI가 지금 무엇을 보고 있는지"를 항상 이해할 수 있게 만들어야 한다.

디자인적으로는 데스크톱에서 우측 고정 레일, 모바일에서는 별도 전환 화면 또는 바텀 내비게이션 연결 화면으로 제공하는 것이 적합하다. 각 아이템은 아이콘, 제목, 타입, 메타데이터, 활성 토글을 함께 가져야 하며, 하단에는 `Context Ready`, `Switch to Chat`, `Switch to Graph` 같은 세션 전환 액션을 둘 수 있어야 한다.

## 10. Agent Chatting 상세 기획

Agent Chatting은 단순한 대화 기록이 아니라 다음 세 기능을 동시에 수행해야 한다.

1. 질문과 생성
2. 기존 지식 회수 및 재조합
3. 대화 결과의 지식화

핵심 요구사항은 다음과 같다.

- 대화 중 특정 메시지를 노드로 저장 가능해야 한다.
- 저장 시 어떤 디렉토리나 주제에 연결할지 결정할 수 있어야 한다.
- 현재 주입된 컨텍스트를 눈에 보이는 형태로 확인할 수 있어야 한다.
- 에이전트 응답이 어떤 컨텍스트를 기반으로 생성되었는지 추적 가능해야 한다.
- 응답 내부에 `Neural Insight`, `Agent Action`, `Saved as Knowledge Node` 같은 구조화된 블록을 삽입할 수 있어야 한다.
- 입력창은 단순 텍스트 필드가 아니라 첨부, 음성, 빠른 액션을 수용하는 옴니바 형태가 적합하다.

## 11. 문서 편집기 기획

문서 편집기 화면은 노드를 클릭해 내부로 진입했을 때 나타나는 상세 화면이다.

### 11.1 역할

- 문서형 노드의 본문 열람
- 마크다운 기반 편집
- 구조화된 문서 작성
- 렌더링과 편집의 유기적 전환

### 11.2 UX 방향

Notion, Obsidian 등에서 제공하는 수준의 마크다운 렌더링 및 편집 경험을 목표로 한다.

- 읽기와 쓰기 전환이 자연스러워야 한다.
- 헤더, 리스트, 체크박스, 코드블록, 인용문, 링크, 임베드 등 기본 블록을 잘 지원해야 한다.
- 문서 내용이 다시 지식 그래프와 컨텍스트 흐름에 연결되어야 한다.

즉, 편집기는 독립된 문서 앱이 아니라 그래프와 채팅 생태계 안의 상세 뷰여야 한다.

### 11.3 편집기 화면 구성 원칙

디자인 시안 기준으로 문서 편집기는 다음 구성을 갖는 것이 적절하다.

- 좌측: 컬렉션 또는 문서 탐색 내비게이션
- 중앙: 넓은 본문 읽기/편집 캔버스
- 우측: 현재 문서와 연결된 Active Context
- 본문 내부 또는 하단: AI 보조 툴바

이는 "문서를 읽는 화면"이 아니라 "지식을 작성하면서 즉시 AI와 연결하는 화면"이라는 점을 반영한다.

## 12. 외부 연동 방향

시냅스노트는 내부 에이전트만 다루는 서비스가 아니라 외부 AI 및 외부 지식 소스와 연결되는 플랫폼을 지향한다.

### 12.1 외부 AI 연동

- ChatGPT
- Claude
- Gemini

외부 챗봇과의 대화 내용도 시냅스노트에 저장하거나 연결 가능해야 한다. 즉, 시냅스노트는 자체 대화 인터페이스이면서 동시에 여러 AI 채널의 지식을 집계하는 허브가 되어야 한다.

### 12.2 외부 지식 소스 연동

- 웹 아티클
- PDF
- Git Issue
- 코드 저장소

외부 자산은 단순 링크 저장이 아니라 노드화되어 그래프와 컨텍스트 주입 흐름 안에 들어와야 한다.

## 13. 기술 방향

### 13.1 프론트엔드

- Next.js / React 기반 반응형 웹
- 시각적 노드 조작 중심 UI
- 모바일/데스크톱 공통 정보 구조 유지

### 13.2 백엔드

- FastAPI
- CouchDB 기반 실시간 자동 동기화
- id/pw 기반 인증

### 13.3 데이터 및 AI

- 사용 가능한 LLM 선택지 제공
  - Claude Code 우선
  - Codex 차순위
  - Gemini 보조
- RAG 파이프라인
  - LlamaIndex
  - Qdrant

### 13.4 확장 인터페이스

- MCP Server
- Skills 기반 외부 연결

목표는 특정 모델이나 특정 서비스에 종속되지 않고, 지식 저장소와 에이전트 오케스트레이션 계층을 분리 가능한 구조로 가져가는 것이다.

### 13.5 디자인 시스템 방향

`docs/design` 시안 기준으로 제품의 비주얼 방향은 `Neural Operating System` 콘셉트에 맞춰 정리되어야 한다.

- 다크 차콜 기반 배경 위에 보라-블루 액센트 사용
- 실선 구분보다 글래스 패널, 간격, 발광을 활용한 계층화
- `Lexend` 중심 헤드라인과 `Inter` 중심 본문 조합
- 좌측 내비게이션, 중앙 워크스페이스, 우측 Active Context Rail의 공통 셸 유지
- 모바일에서는 `Chat / Context / Graph` 흐름을 보존하는 전환 구조 채택

상세 디자인 규칙은 `docs/design/design_specification.md`를 기준 문서로 삼는다.

## 14. 벤치마크

- Obsidian / Notion: 지식 관리와 문서 편집
- ChatGPT / Claude: 대화형 AI 인터페이스
- NotebookLM / Perplexity: 검색 및 컨텍스트 기반 응답
- Second Brain 워크플로우: 개인 지식 운영 철학

시냅스노트는 이들을 단순 조합하는 것이 아니라, "대화에서 생성된 지식이 그래프로 축적되고 다시 컨텍스트로 사용되는 순환 구조"를 제품 중심 가치로 삼는다.

## 15. MVP 범위 제안

### 15.1 반드시 포함할 기능

1. Chat / Context Manager / Knowledge Base 3축 화면 구조
2. Knowledge Base의 `Library View`와 `Knowledge Graph View` 동시 제공
3. 그래프에서 노드 선택 후 컨텍스트 주입
4. 채팅 메시지의 노드화
5. 디렉토리 구조 기반 그래프 탐색
6. 문서형 노드 상세 편집 화면
7. 우측 `Active Context` 레일 또는 동등한 컨텍스트 패널 구조
8. 모바일 스와이프 또는 탭 기반 화면 전환

### 15.2 후속 확장 기능

1. 학습 모드
2. 외부 챗봇 연동 자동 저장
3. Git Issue, PDF, Web Article, Code 전용 노드 강화
4. 의미 기반 추천 연결
5. 고급 RAG 및 에이전트 오케스트레이션

## 16. 성공 기준

초기 제품은 다음 질문에 긍정적으로 답할 수 있어야 한다.

1. 사용자가 AI와 대화한 내용을 쉽게 저장할 수 있는가
2. 저장된 지식을 다시 대화에 유의미하게 주입할 수 있는가
3. 사용자가 현재 컨텍스트를 직관적으로 이해하고 통제할 수 있는가
4. 그래프 탐색이 단순 시각화가 아니라 실제 생산성 향상으로 이어지는가
5. 모바일에서도 핵심 흐름이 무너지지 않는가

## 17. 결론

시냅스노트의 본질은 AI 채팅, 컨텍스트 제어, Knowledge Base, 문서 편집을 하나의 순환 구조로 통합하는 데 있다.

대화는 지식을 만들고, 지식은 Knowledge Base에 축적되며, 그중 `Knowledge Graph View`는 관계를 탐색하는 시각적 인터페이스가 된다. Context Manager는 이 순환의 중심 허브이며, 모바일과 데스크톱 모두에서 이 구조가 끊기지 않도록 설계되어야 한다.

따라서 시냅스노트는 "노트 앱에 AI를 붙인 서비스"가 아니라, "대화가 사라지지 않고 누적되는 개인 지식 운영체제"로 정의되어야 한다.
